# Prompt 15 — Independent `n_count` verification; MSR action computation

## Context

Read `./.documents/onion_model_planning.md` and `./.documents/onion_notes.tex`
before starting — §"The MSR action" (`eq:msr-action`), §"Noise statistics
in $y$ coordinates" (`eq:ncount`, `eq:Dnoise-diag`, `eq:Dnoise-cross`).

This prompt has two parts. Part A closes a real test-coverage gap found
while planning Part B — every existing test touching `n_count`
(`diluted_diffusion_coefficients`, `noise_source_terms`, and the
`disable_spatial_coupling`-vs-`FullInstanton` reduction test) constructs
its own reference value using the *same* `n_count` formula the production
code uses, so a wrong exponent, prefactor, or node-indexing in `eq:ncount`
would cancel identically on both sides and never be caught. This was
confirmed by direct investigation, not assumed — checking the existing
fixture showed $n_{\rm count}$ genuinely spans two orders of magnitude
across nodes there (not pinned to some special value), but the *match*
comes from algebraic construction (response-field splines pre-multiplied
by the same `n_count` array the production code then divides by), not from
`n_count` being independently verified.

## Task

### Part A — independent `n_count` check

If `n_count`'s computation isn't already a standalone, directly-callable
function (check `forward_rhs.py`'s `diluted_diffusion_coefficients` — it
may currently compute `n_count` inline rather than exposing it separately),
factor it out into its own function first, purely additively, reused by
`diluted_diffusion_coefficients`. This mirrors the pattern already used for
`noise_source_terms`/`diluted_diffusion_coefficients` themselves — pull a
piece out so it's independently testable, no behavior change.

New test: pick **hand-chosen** `delta_s_N` and `delta_s_loc` values (plain
numbers, not generated by calling `delta_s()` — the point is to avoid the
same reference/production coupling that caused this gap) for a small
number of nodes with distinct $y_j$, and compare the production `n_count`
function's output against `eq:ncount`'s closed form
($\tfrac32\Delta s(N)\,e^{3\Delta s_{\rm loc}(y,N)}\,e^{-\frac32(y+1)\Delta s(N)}$)
evaluated independently in the test using plain Python arithmetic — not by
calling any shared helper the production code also uses. This is
specifically the check the existing tests can't do.

### Part B — MSR action

**The on-shell integrand is the full quadratic form**, confirmed directly
from `eq:msr-action`: on the saddle point (forward EOM exactly satisfied,
linear-in-response-field terms vanish identically),

$$S = -\int_{N_{\rm init}}^{N_{\rm final}}dN\int_{-1}^1\mu(y,N)\,dy\left[\frac{D_\phi}{2}\rfield^2 + D_{\phi\pi}\rfield\rmom + \frac{D_\mom}{2}\rmom^2\right]$$

**not** just the single `D11*P1**2` term `FullInstanton`'s own code
computes. Checked why that isn't a discrepancy: `FullInstanton`'s formula
is correct *for the only diffusion model currently implemented*
(`MasslessDecoupledDiffusion.D_matrix` returns `(D11, 0.0, 0.0)` —
$D_{12}=D_{22}\equiv0$ exactly there), so the missing terms are identically
zero in that case, not dropped incorrectly. Implement the **full**
three-term formula here regardless — it reduces exactly to `FullInstanton`'s
expression for this diffusion model (a valid, exact reduction test), and
won't silently lose terms for a future diffusion model with genuine
off-diagonal structure. Document this reasoning in the new code so a
future editor doesn't "simplify" it back down to one term by analogy with
`FullInstanton`.

**Quadrature strategy, settled in discussion**:

- **$y$-direction**: reuse `grid.weights` directly (LGL quadrature,
  spectrally accurate, no new scheme) — $\int_{-1}^1\mu\,f\,dy\approx\sum_j
  w_j\,\mu(y_j,N)\,f(y_j,N)$, a weighted dot product against the diluted
  quadratic-form integrand computed at every node.
- **$N$-direction**: `np.trapezoid` over `solve_picard`'s own dense
  $N$-grid — mirroring `FullInstanton`'s own `np.trapezoid(D11_arr *
  P1_f**2, N_grid)` call as closely as possible, for consistency and for a
  clean reduction-limit comparison, not a higher-order or resampled scheme.
- Combine as a two-stage, vectorized reduction: build the full
  $(N_{\rm grid\ size},\,n_{\rm collocation\ points})$ integrand array
  once (needs `phi_grid`, `pi_grid`, `rfield_grid`, `rmom_grid`, and the
  diluted $D_\phi,D_{\phi\pi},D_\mom$ at **every** row of the dense solver
  grid, not just the final row — recompute `delta_s_N`/`delta_s_loc_array`
  per row from that row's own core/per-node state, reusing `delta_s()`),
  contract the node axis via the `grid.weights`-and-$\mu$ product to get a
  1D array over $N$, then `np.trapezoid` that against `N_grid`.
- **Empirical $N$-quadrature convergence check**: trapezoid is only
  second-order accurate, unlike the spectrally-accurate $y$-direction, so
  add a diagnostic test comparing `msr_action` computed on `solve_picard`'s
  native grid against a resampled finer grid (via the dense-output
  interpolant already available from the underlying `solve_ivp` calls —
  reuse it, don't re-solve) — confirming the discrepancy shrinks as
  expected for trapezoidal convergence, same philosophy as the empirically-
  tuned quadrature order back in `Numerics/LGLCollocation.py`. This is a
  diagnostic, not a strict pass/fail gate — report the convergence rate
  found, don't assert a specific tight tolerance without first seeing what
  it actually is.

Wire the result into `GradientCoupledInstanton.compute()`, populating the
`msr_action` column (left `nullable=True`/unpopulated in prompt 14).

## Tests

- Part A: the independent `n_count` check described above.
- Part B:
  - Reduction-limit test: `msr_action` computed via the full three-term
    formula, gradient coupling disabled, compared against `FullInstanton`'s
    own `msr_action` for the same underlying trajectory/potential/
    diffusion model — should match to numerical precision (both terms that
    `FullInstanton` omits are exactly zero for `MasslessDecoupledDiffusion`,
    so this is an exact check, not an approximate one).
  - The empirical convergence diagnostic described above.
  - A direct unit test of the quadrature contraction itself (LGL weights ×
    measure × integrand, summed over nodes) against a synthetic integrand
    with a known closed-form $y$-integral, independent of any physics
    inputs — isolating "is the quadrature applied correctly" from "is the
    physics integrand correct."
